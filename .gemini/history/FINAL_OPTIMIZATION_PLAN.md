## ✅ Признанные ошибки в предыдущем анализе

### 1. Локальный билд ≠ улучшение PSI
- **Было:** "Ожидаемый результат: PSI Mobile 66 → 70-72"
- **Правильно:** Локальный билд - это **инженерная гигиена**, не улучшение метрик
- **Реальность:** SSG может дать +1-2 поинта, но не +4-6

### 2. Lazy Loading (уточнение)
- **Было:** "Не делать lazy loading Dashboard — ухудшит LCP"
- **Правильно:** 
  - **Лендинг (`/`)** - НЕ lazy load (синхронный, максимально легкий)
  - **Приложение (`/app/*`)** - lazy load целого блока с провайдерами (Query/Supabase)
  - Нельзя lazy load сам первый экран (hero/landing) - это убьет LCP

### 3. Проверка лендинга за 30 минут
- **Было:** "Проверить Landing (30 минут)"
- **Правильно:** Нужен **bundle-анализ** перед оптимизацией
- **Реальность:** Без анализа нельзя понять, что именно грузится

### Этап 1: Bundle-анализ (ОБЯЗАТЕЛЬНО ПЕРВЫМ)

#### 1.1. Настроить bundle-анализатор

**Что сделать:**

1. Убедиться, что `rollup-plugin-visualizer` уже подключен (он есть в `vite.config.ts`)

2. Запустить анализ:
   ```bash
   npm run build:analyze
   ```

3. Открыть `dist/stats.html` в браузере

4. Посмотреть:
   - Что входит в `index.js` (initial chunk)
   - Что входит в `vendor.js`
   - Размеры каждого chunk

**Вопросы для анализа:**
- ✅ Есть ли Supabase в initial chunk?
- ✅ Есть ли React Query в initial chunk?
- ✅ Есть ли Radix UI в initial chunk?
- ✅ Какой размер initial bundle для `/` (лендинг)?

**Ожидаемый результат:**
- Понимание, что именно грузится на лендинге
- Данные для принятия решений

**✅ РЕЗУЛЬТАТЫ АНАЛИЗА (выполнено):**
- `vendor.js`: **1,020.46 kB** (314.44 kB gzipped) - огромный chunk!
- `index.js`: **302.82 kB** (102.76 kB gzipped)
- **Итого initial:** ~417 KB gzipped (vendor + index)
- **Проблема подтверждена:** vendor.js содержит все зависимости (Supabase, Query, Radix UI)

### Этап 2: Локальный билд (инженерная гигиена)

#### 2.1. Создать скрипт для деплоя

**Что сделать:**

1. Добавить в `package.json`:
   ```json
   {
     "scripts": {
       "deploy:prod": "npm run build && npm run prerender && vercel deploy dist --prod"
     }
   }
   ```
   
   **Важно:** Флаг `--prebuilt` НЕ нужен при деплое `dist`. Он нужен только для `.vercel/output`.

2. Убедиться, что Vercel не делает свой билд:
   - В настройках Vercel проекта отключить автодеплой из Git
   - Или использовать `vercel.json` с настройками

**Что это даст:**
- ✅ Решает проблему с Chrome на Vercel
- ✅ Стабильность деплоя (SSG не "отвалится")
- ✅ Контроль над артефактами
- ⚠️ **НЕ даст значительного прироста PSI** (может +1-2 поинта)

**Ожидаемый результат:**
- Стабильный деплой без проблем с Chrome
- PSI Mobile: 66 → 67-68 (небольшой прирост от SSG)

### Этап 3: Оптимизация на основе bundle-анализа

#### 3.1. Если Supabase/Query/Radix в initial chunk

**Проблема:**
- Даже если `Index` lazy loaded, зависимости могут грузиться в `App.tsx`
- `QueryClient`, `PersistQueryClientProvider` импортируются синхронно

**Решение:**

1. **Lazy load Query Provider:**
   ```typescript
   // App.tsx
   const QueryProvider = lazy(() => 
     import("@/providers/QueryProvider").then(m => ({ default: m.QueryProvider }))
   );
   ```

2. **Lazy load Supabase client:**
   ```typescript
   // Создать отдельный провайдер для Supabase
   const SupabaseProvider = lazy(() => 
     import("@/providers/SupabaseProvider").then(m => ({ default: m.SupabaseProvider }))
   );
   ```

3. **Условная загрузка:**
   - На лендинге (`/`) - не грузить Query/Supabase
   - На `/app` или после авторизации - грузить

**Ожидаемый результат:**
- Initial bundle для лендинга: 417KB → 150-200KB (если удастся вынести Supabase/Query/Radix)
- PSI Mobile для лендинга: 66 → 75-80
- **Важно:** Это возможно только если Supabase/Query/Radix действительно "переедут" в отдельный chunk

### Этап 4: Разделение лендинга и приложения (если нужен 90+)

#### 4.1. Создать отдельный легкий лендинг

**Вариант A: Два отдельных фронта (рекомендуется)**

1. **Создать `landing/` проект:**
   ```
   landing/
     ├── src/
     │   ├── main.tsx
     │   ├── LandingApp.tsx
     │   └── components/
     ├── index.html
     ├── vite.config.ts
     └── package.json
   ```

2. **Минимальные зависимости:**
   - Только React
   - Без Supabase
   - Без React Query
   - Без Radix UI
   - Минимум компонентов

3. **Деплой:**
   - `landing` → `skilyapp.com/`
   - `app` → `skilyapp.com/app` или `app.skilyapp.com`

**Вариант B: Один репо, два entrypoint'а**

1. Настроить Vite для двух entry points:
   ```typescript
   // vite.config.ts
   build: {
     rollupOptions: {
       input: {
         main: './index.html',
         landing: './landing.html'
       }
     }
   }
   ```

2. Создать `landing.html` и `landing/main.tsx`

**Ожидаемый результат:**
- Лендинг: ~150KB gzipped
- PSI Mobile для лендинга: 85-90+
- Приложение остается на 66-70 (это нормально)

## 🔍 ТЕХНИЧЕСКИЕ ДЕТАЛИ

### Текущая структура роутинга

**Как сейчас:**
- `App.tsx` - главный компонент, импортирует Query/Supabase синхронно
- `Index.tsx` - lazy loaded, но зависимости уже в `App.tsx`
- `Landing.tsx` - компонент внутри `Index.tsx`

**Проблема:**
- Даже если `Index` lazy, `App.tsx` уже загрузил Query/Supabase
- Initial bundle содержит все зависимости

**Решение:**
- Вынести Query/Supabase в lazy providers
- Условная загрузка только для авторизованных пользователей
