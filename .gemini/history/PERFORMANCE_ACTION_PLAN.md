## ✅ Что проверить ПЕРВЫМ

### 1. Skeleton в production
**Вопрос**: Задеплоен ли skeleton в production?

**Проверка**:
```bash
# 1. Проверить production HTML
curl https://skilyapp.com/ | grep -A 20 "app-skeleton"

# 2. Проверить filmstrip в PageSpeed
# Открыть отчёт → Performance → Filmstrip
# Виден ли skeleton в первые секунды?

# 3. DevTools Performance
# Записать профиль загрузки → проверить когда появляется skeleton
```

**Если skeleton НЕ в prod**:
- Задеплоить изменения
- Запустить новый PageSpeed анализ
- Сравнить метрики

**Если skeleton в prod, но FCP всё равно 6.4s**:

⚠️ **Критично проверить Render-Blocking CSS:**

Если в PageSpeed Filmstrip первые кадры **белые**, а не серые (skeleton), проблема может быть в:

1. **CSS skeleton во внешнем файле** (render-blocking)
   - ❌ Плохо: `<link rel="stylesheet" href="skeleton.css">`
   - ✅ Хорошо: CSS inline в `<style>` теге (у нас так и есть)

2. **Skeleton скрыт стилями**
   - Проверить что нет `display:none` или `opacity:0` на `.app-skeleton`
   - Проверить что skeleton виден ДО загрузки JS

3. **Network waterfall**
   - Проверить что HTML загружается быстро (TTFB)
   - Проверить что нет блокирующих CSS файлов до skeleton

### 2. Network Waterfall анализ
**Цель**: Найти что реально блокирует FCP

**Как проверить**:
```bash
# Chrome DevTools:
1. Network tab → Throttling: Slow 4G
2. Reload страницу
3. Посмотреть waterfall:
   - Какие ресурсы грузятся до FCP?
   - Что блокирует рендеринг?
   - Какие запросы самые медленные?
```

**Как читать waterfall (подсказки от Димы)**:

- **Синяя полоска (HTML)**: Если длинная → сервер долго думает (TTFB проблема)
- **Зелёные полоски (JS/CSS)**: Если начинаются после HTML, но до FCP → это проблема
- **Размер**: Если `index-*.js` размером 800KB+ → это "убийца" FCP на мобильном интернете

⚠️ **Важно**: Цвета в waterfall DevTools могут отличаться. Главное - смотреть:
- **Тип ресурса** (Doc/CSS/Script/Image)
- **Факт**, что ресурс грузится до FCP
- **Размер** ресурса
- **Последовательность** загрузки

**Что искать**:
- HTML загружается быстро? (TTFB < 200ms)
- CSS блокирует? (render-blocking - должно быть inline для skeleton)
- JS chunks грузятся последовательно? (должны параллельно через HTTP/2)
- Изображения блокируют? (не должны быть до FCP)
- API запросы до FCP? (не должны быть - FCP не ждёт API)
- Долгий TLS handshake? (проверить preconnect)
- Редиректы? (увеличивают время загрузки)

**Дополнительно в PageSpeed**:
- Секция **"Opportunities"** - показывает render-blocking ресурсы
- Секция **"Diagnostics"** - показывает "Reduce unused JavaScript/CSS"
- **LCP Element** - какой элемент признан LCP (hero-картинка, блок текста, карточка)

### 3. Bundle Analysis
**Цель**: Понять размер и состав bundle

**Как запустить**:
```bash
npm run build:analyze
# Откроется визуализация bundle (stats.html)
```

**Что проверить**:
- Размер vendor chunk
- Размер index chunk
- Самые тяжёлые зависимости
- Unused JS (через Coverage в DevTools)
- **Initial vs Async chunks**: Реально ли `charts` не входит в initial load?
- **Размеры новых chunks**: `react-vendor`, `ui-vendor`, `data-vendor` (конкретные KB/MB)
- **Кандидаты на вынос**: Нет ли одной-двух зависимостей которые "жрут" >30-40% любого chunk?

**Ожидаемые проблемы**:
- recharts (если используется только в админке) - должен быть в async chunk
- framer-motion (если используется везде) - проверить размер
- Большой vendor bundle (все node_modules вместе) - теперь разделён

### Шаг 2: Исправления (после диагностики)

#### Если skeleton НЕ в prod:
- Задеплоить изменения
- Перезапустить PageSpeed
- Сравнить метрики

#### Если проблема в Network:
- Preload критичных ресурсов (через Vite plugin)
- Preconnect к Supabase (уже есть)
- Оптимизация последовательной загрузки

#### Если проблема в Bundle:
- ✅ **Code splitting** - улучшенный конфиг уже в `vite.config.ts`:
  - `react-vendor` (React, ReactDOM, React Router)
  - `ui-vendor` (framer-motion, @mui)
  - `charts` (recharts - только для админки)
  - `data-vendor` (Supabase, React Query)
  - `vendor` (остальное)
  
  **Что это даёт:**
  - Браузер качает chunks **параллельно** (HTTP/2)
  - Основной `index.js` становится крошечным
  - Запускается быстрее
  - **Главный профит**: `charts` не грузится на обычных страницах (только админка)
  
  ⚠️ **Важные нюансы:**
  - HTTP/2 параллельная загрузка **не серебряная пуля** - общие байты никуда не делись
  - Если на первом экране нужны все chunks (`react-vendor`, `ui-vendor`, `data-vendor`, `vendor`), FCP всё равно упирается в их **суммарный размер**
  - **Реальный выигрыш** - от отложенной загрузки неиспользуемых частей (recharts для админки)
  - Не дробить на слишком много чанков (4-5 - разумно, больше - может навредить)
  
  **В bundle analysis проверить:**
  - Реально ли `charts` **не входит** в initial load (см. `initial` vs `async` chunks)
  - Размеры `react-vendor`, `ui-vendor`, `data-vendor` (конкретные KB/MB)
  - Нет ли одной-двух зависимостей которые "жрут" >30-40% любого из них
  
- Lazy loading тяжёлых компонентов (уже есть)
- Удаление неиспользуемых зависимостей
- Проверить Coverage в DevTools (unused JS)

#### Если проблема в API (LCP):
- React Query prefetchQuery (не `<link rel="prefetch">`)
- Параллелизация запросов
- Агрессивное кэширование
- SSR для критичных данных (если возможно)

#### Если проблема в изображениях:
- WebP формат
- Lazy loading
- Responsive images (srcset)
- CDN для изображений

### Шаг 3: Оптимизация (после исправлений)

1. **Оптимизация изображений**
   - WebP конвертация
   - Lazy loading
   - Responsive images

2. **Preload критичных ресурсов**
   - Через Vite plugin (не вручную)
   - Только критичные chunks

3. **Оптимизация API**
   - Prefetch через React Query
   - Параллелизация
   - Кэширование

## ❌ Что НЕ делать

### 1. HTTP/2 Server Push
- ❌ Устарел, не рекомендуется
- ✅ Использовать `preload` вместо push

### 2. Service Worker для первой загрузки
- ❌ SW не ускоряет первую загрузку
- ✅ SW помогает только для повторных визитов

### 3. `<link rel="prefetch" href="/api/dashboard">`
- ❌ Не самый надёжный паттерн
- ✅ Использовать React Query `prefetchQuery` в JS

### 4. Preload с `*` в HTML
- ❌ `*` не подставится сам
- ✅ Использовать Vite plugin для автоматической подстановки
