## 🔄 Рабочий процесс

### ⚡ Важно про Git и ветки:

**✅ Работа с одной веткой в двух редакторах:**
- Оба редактора (Cursor и Antigravity) используют **ОДИН Git репозиторий**
- Работаешь с **ОДНОЙ веткой** (например, `main`) в обоих редакторах
- **НЕ нужно создавать отдельные ветки для разных редакторов**
- Git видит все изменения независимо от того, в каком редакторе ты работаешь
- Коммить можно из любого редактора - все идет в одну ветку

**Пример:**
```bash
# В Cursor работаешь над файлом A, коммитишь
git add src/components/A.tsx
git commit -m "feat: update component A"

# Переключаешься в Antigravity, работаешь над файлом B
git add src/components/B.tsx
git commit -m "feat: update component B"

# Оба коммита в одной ветке (main)
```

### Перед началом работы:

1. **Проверь текущую ветку:**
   ```bash
   git branch
   ```

2. **Создай новую ветку для фичи (если нужно):**
   ```bash
   git checkout -b feature/название-фичи
   ```
   **Примечание:** Это для изоляции фичи, а не для разделения редакторов!

3. **Убедись, что у тебя последняя версия:**
   ```bash
   git pull origin main
   ```

### Во время работы:

1. **Регулярно коммить изменения:**
   ```bash
   git add .
   git commit -m "feat: описание изменений"
   ```

2. **Проверяй линтер:**
   ```bash
   npm run lint
   ```

3. **Тестируй локально:**
   ```bash
   npm run dev
   ```

### После завершения работы:

1. **Проверь статус (важно при работе в двух редакторах!):**
   ```bash
   git status
   git diff
   ```
   Убедись, что видишь все изменения из обоих редакторов

2. **Запушь изменения:**
   ```bash
   # Если работаешь в feature ветке
   git push origin feature/название-фичи
   
   # Или если работаешь в main
   git push origin main
   ```

3. **Создай Pull Request в GitHub** (если работаешь в feature ветке)

4. **Или смержь в main и запушь:**
   ```bash
   git checkout main
   git merge feature/название-фичи
   git push origin main
   ```

5. **Автоматический деплой в Vercel** произойдет после пуша в main

**💡 Важно:** Неважно, из какого редактора ты пушишь - все идет в одну ветку на GitHub!

## 🗄️ Оптимизация Supabase (критично)

### Правила работы с Supabase:

1. **Минимизируй количество запросов**
2. **Никогда не используй real-time**, если можно обойтись polling/edge-триггером
3. **Любая функция Supabase должна:**
   - работать < 300 мс
   - иметь предохранители
   - не вызывать каскадные запросы

4. **Любая миграция должна быть оптимизирована:**
   - не создавать лишние индексы
   - не хранить дубликаты
   - использовать SMALLINT/INTEGER где возможно

5. **Всегда проверяй, можно ли объединить несколько запросов в один RPC**

6. **Используй `select('...',{ count: 'exact', head: true })` вместо полноценного запроса**, если нужно только проверить существование

## 🐛 Решение проблем

### Проблема: "Module not found"

```bash
# Удали node_modules и переустанови
rm -rf node_modules package-lock.json
npm install
```

### Проблема: "Port already in use"

```bash
# Найди процесс на порту 8080
lsof -ti:8080 | xargs kill -9

# Или измени порт в vite.config.ts
```

### Проблема: "Build failed"

```bash
# Очисти кеш и пересобери
rm -rf dist .vite
npm run build
```

3. **Запусти проект и проверь подключение:**
   ```bash
   npm run dev
   ```
   - Открой в браузере
   - Проверь, что данные загружаются из Supabase

### ⚠️ Если что-то не работает:

1. **Проверь путь к .env.local:**
   - Должен быть в корне проекта: `/Users/dimka/Desktop/Sdadim/sdadim-dgt-prep/.env.local`

2. **Проверь формат переменных:**
   ```env
   VITE_SUPABASE_URL=https://...
   VITE_SUPABASE_ANON_KEY=eyJ...
   ```

   - Без пробелов вокруг `=`

3. **Перезапусти dev сервер:**
