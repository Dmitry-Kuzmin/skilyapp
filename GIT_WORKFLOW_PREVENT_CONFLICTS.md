# 🔄 Правильный Git Workflow для предотвращения конфликтов

## 📋 Основные правила

### ✅ ПРАВИЛО №1: Всегда синхронизируйся ПЕРЕД началом работы

**Перед началом работы в ЛЮБОМ редакторе:**

```bash
# 1. Сохрани свои изменения (если есть)
git add .
git commit -m "wip: промежуточные изменения"

# 2. Получи последние изменения с сервера
git fetch origin

# 3. Синхронизируйся с удалённой веткой
git pull origin main

# 4. Только теперь начинай работать!
```

### ✅ ПРАВИЛО №2: Коммить и пушить часто

**Не накапливай изменения!**

```bash
# После каждой логической единицы работы:
git add .
git commit -m "feat: добавил новую функцию X"
git push origin main
```

**Почему это важно:**
- Чем чаще пушишь → тем меньше вероятность конфликтов
- Другие редакторы видят твои изменения сразу
- Легче откатить, если что-то пошло не так

### ✅ ПРАВИЛО №3: Используй одну ветку для работы

**Если работаешь один:**
- Используй `main` ветку
- Не создавай лишних веток без необходимости

**Если работаешь в команде:**
- Создавай feature-ветки для каждой задачи
- Сливай через Pull Request

## 🛠️ Настройка Git для автоматического разрешения

### Вариант 1: Автоматический rebase (рекомендуется)

```bash
# Настрой Git на автоматический rebase при pull
git config --global pull.rebase true
```

**Что это даёт:**
- При `git pull` твои локальные коммиты будут "перемещены" поверх удалённых
- Линейная история без merge-коммитов
- Меньше конфликтов

### Вариант 2: Автоматический merge (проще, но больше merge-коммитов)

```bash
# Настрой Git на автоматический merge
git config --global pull.ff only
```

**Что это даёт:**
- Git попытается сделать fast-forward merge
- Если не получится — предупредит о конфликтах

### Вариант 3: Стратегия merge (для сложных случаев)

```bash
# Настрой стратегию merge (берёт версию из удалённой ветки при конфликтах)
git config --global merge.ours.driver true
git config --global pull.twohead ours
```

## 📝 Пошаговая инструкция для работы в разных редакторах

### Сценарий: Работаешь в Cursor, потом в VS Code, потом снова в Cursor

#### ШАГ 1: Перед закрытием Cursor

```bash
# 1. Проверь статус
git status

# 2. Если есть изменения — закоммить
git add .
git commit -m "feat: описание изменений"
git push origin main

# 3. Убедись, что всё запушено
git status  # должно быть "nothing to commit, working tree clean"
```

#### ШАГ 2: Открываешь VS Code (или другой редактор)

```bash
# 1. Сначала синхронизируйся
git fetch origin
git pull origin main

# 2. Теперь работай
# ... делаешь изменения ...

# 3. Перед закрытием — коммить и пушить
git add .
git commit -m "feat: описание изменений"
git push origin main
```

#### ШАГ 3: Возвращаешься в Cursor

```bash
# 1. ОБЯЗАТЕЛЬНО синхронизируйся перед работой!
git fetch origin
git pull origin main

# 2. Теперь работай
```

## 🎯 Автоматизация: Создай алиасы

### Добавь в `~/.zshrc` или `~/.bashrc`:

```bash
# Быстрая синхронизация
alias gsync='git fetch origin && git pull origin main'

# Коммит и пуш одной командой
alias gpush='git add . && git commit -m "$1" && git push origin main'

# Проверка статуса перед работой
alias gcheck='git fetch origin && git status && echo "---" && git log --oneline origin/main..HEAD'
```

**Использование:**
```bash
# Перед началом работы
gsync

# После работы
gpush "feat: добавил новую функцию"
```

## ⚠️ Что делать, если конфликт всё-таки возник

### Вариант 1: Автоматическое разрешение (берёт версию из удалённой ветки)

```bash
git pull origin main
# Если конфликт:
git checkout --theirs .  # берёт версию из удалённой ветки
git add .
git commit -m "merge: разрешены конфликты"
```

### Вариант 2: Ручное разрешение (если нужно сохранить оба варианта)

```bash
git pull origin main
# Открой файлы с конфликтами, разреши вручную
# Удали маркеры <<<<<<< HEAD, =======, >>>>>>> origin/main
git add .
git commit -m "merge: разрешены конфликты вручную"
```

### Вариант 3: Отменить merge и начать заново

```bash
git merge --abort  # отменяет merge
git pull origin main  # попробуй снова
```

## 🔧 Настройка для твоего проекта

### Рекомендуемые настройки:

```bash
# 1. Автоматический rebase при pull
git config pull.rebase true

# 2. Автоматический prune (удаление удалённых веток)
git config fetch.prune true

# 3. Показывать статус после pull
git config pull.rebase true
git config rebase.autoStash true  # автоматически сохраняет незакоммиченные изменения

# 4. Цветной вывод
git config color.ui auto
```

## 🎓 Лучшие практики

### ✅ ДЕЛАЙ:
1. **Коммить часто** — после каждой логической единицы работы
2. **Пушить часто** — не накапливай изменения
3. **Синхронизироваться перед работой** — всегда!
4. **Писать понятные сообщения коммитов** — легче понять, что изменилось

### ❌ НЕ ДЕЛАЙ:
1. **Не работай без синхронизации** — всегда `git pull` перед началом
2. **Не накапливай изменения** — коммить и пушить регулярно
3. **Не игнорируй конфликты** — разрешай сразу
4. **Не коммить всё подряд** — группируй логически связанные изменения

## 🚀 Быстрый старт

### Скопируй и выполни эти команды один раз:

```bash
# Настрой Git для удобной работы
git config pull.rebase true
git config fetch.prune true
git config rebase.autoStash true

# Создай алиасы (добавь в ~/.zshrc)
echo 'alias gsync="git fetch origin && git pull origin main"' >> ~/.zshrc
echo 'alias gcheck="git fetch origin && git status"' >> ~/.zshrc
source ~/.zshrc
```

### Теперь используй:

```bash
# Перед началом работы (в ЛЮБОМ редакторе)
gsync

# После работы
git add . && git commit -m "описание" && git push origin main
```
