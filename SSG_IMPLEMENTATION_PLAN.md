# План внедрения SSG (Static Site Generation) для SEO

## 🎯 Цель

Внедрить SSG для публичных страниц (Blog, Article, Landing) для улучшения SEO без сложностей SSR. Приложение остаётся SPA.

## ✅ Почему SSG, а не SSR?

### Проблемы SSR (которые мы избегаем):
- ❌ Нужен Node.js сервер (дополнительные расходы)
- ❌ Проблемы с `window`, `document`, `Telegram.WebApp` на сервере
- ❌ Библиотеки могут не работать в SSR
- ❌ Сложность внедрения на существующем проекте
- ❌ Риск сломать Telegram WebApp интеграцию

### Преимущества SSG:
- ✅ Не нужен сервер — просто статические HTML файлы
- ✅ Быстрее чем SSR (сервер не "думает", просто отдаёт файл)
- ✅ Безопаснее (нет проблем с `window` на сервере)
- ✅ SEO работает идеально (Google видит полный HTML)
- ✅ Не сломает Telegram WebApp (генерируется на билде, не в рантайме)
- ✅ Дешевле (статический хостинг дешевле Node.js сервера)

## 📋 Страницы для SSG

### ✅ SSG (публичные, SEO-критичные):
- `/` (Landing) — лендинг для неавторизованных
- `/blog` — список статей
- `/article/:slug` — отдельные статьи

### ❌ SPA (приложение, остаётся как есть):
- `/dashboard` — дашборд
- `/tests`, `/games` — тесты и игры
- `/learning` — обучение
- Все остальные страницы приложения

## 🛠️ Реализация

### Инструменты:

**Вариант 1: `vite-plugin-ssg` (рекомендуется)**
- Специально для Vite
- Простая настройка
- Генерирует статические HTML во время билда

**Вариант 2: `prerender-spa-plugin`**
- Универсальный плагин
- Работает с Vite через адаптер

**Вариант 3: Кастомный скрипт**
- Используем Puppeteer/Playwright для рендеринга
- Полный контроль, но больше кода

## 🚀 План реализации (Вариант 1: vite-plugin-ssg)

### Этап 1: Установка и настройка (1 день)

```bash
npm install -D vite-plugin-ssg
```

**vite.config.ts:**
```typescript
import { ssg } from 'vite-plugin-ssg';

export default defineConfig({
  plugins: [
    react(),
    ssg({
      // Указываем только публичные роуты для SSG
      includedRoutes: async () => {
        return [
          '/',
          '/blog',
          // Генерируем статические страницы для всех статей
          ...articles.map(article => `/article/${article.slug}`)
        ];
      },
      // Остальные роуты остаются SPA
    }),
  ],
});
```

## 🔧 Технические детали

### Структура после билда:

```
dist/
  index.html          # Landing (SSG)
  blog.html           # Blog (SSG)
  article/
    novye-voprosy-dgt-2025.html  # Article (SSG)
    top-10-oshibok-na-ekzamene-dgt.html  # Article (SSG)
  assets/             # JS/CSS (как обычно)
  ...                 # Остальные страницы (SPA)
```

### Vercel конфигурация:

```json
{
  "rewrites": [
    // SSG страницы отдаются как статика
    { "source": "/blog", "destination": "/blog.html" },
    { "source": "/article/:slug", "destination": "/article/:slug.html" },
    // Остальные роуты → SPA
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

### Обработка динамических данных:

Если статьи хранятся в базе данных:
1. Во время билда делаем запрос к Supabase
2. Генерируем статические HTML для всех статей
3. При добавлении новой статьи — пересборка (или автоматический деплой)

Если статьи в коде (как сейчас):
- Просто генерируем HTML для всех статей из массива

## ⚠️ Важные моменты

1. **Telegram WebApp API** — не проблема, так как SSG генерируется на билде, а не в рантайме
2. **Мета-теги** — должны быть в компонентах, они попадут в статический HTML
3. **Structured Data** — должны быть в компонентах
4. **Изображения** — нужно оптимизировать (WebP, lazy loading)
5. **Обновление контента** — при добавлении новой статьи нужна пересборка
