# План внедрения SSG (Static Site Generation) для SEO

## 🎯 Цель

Внедрить SSG для публичных страниц (Blog, Article, Landing) для улучшения SEO без сложностей SSR. Приложение остаётся SPA.

## ✅ Почему SSG, а не SSR?

### Проблемы SSR (которые мы избегаем):
- ❌ Нужен Node.js сервер (дополнительные расходы)
- ❌ Проблемы с `window`, `document`, `Telegram.WebApp` на сервере
- ❌ Библиотеки могут не работать в SSR
- ❌ Сложность внедрения на существующем проекте
- ❌ Риск сломать Telegram WebApp интеграцию

### Преимущества SSG:
- ✅ Не нужен сервер — просто статические HTML файлы
- ✅ Быстрее чем SSR (сервер не "думает", просто отдаёт файл)
- ✅ Безопаснее (нет проблем с `window` на сервере)
- ✅ SEO работает идеально (Google видит полный HTML)
- ✅ Не сломает Telegram WebApp (генерируется на билде, не в рантайме)
- ✅ Дешевле (статический хостинг дешевле Node.js сервера)

## 📋 Страницы для SSG

### ✅ SSG (публичные, SEO-критичные):
- `/` (Landing) — лендинг для неавторизованных
- `/blog` — список статей
- `/article/:slug` — отдельные статьи

### ❌ SPA (приложение, остаётся как есть):
- `/dashboard` — дашборд
- `/tests`, `/games` — тесты и игры
- `/learning` — обучение
- Все остальные страницы приложения

## 🛠️ Реализация

### Инструменты:

**Вариант 1: `vite-plugin-ssg` (попробовать сначала)**
- Специально для Vite
- Простая настройка
- Генерирует статические HTML во время билда
- ⚠️ **Нюанс:** Изначально создавался для Vue.js. Для React может требовать адаптер (`vite-react-ssg`) или танцы с бубном

**Вариант 2: `vite-plugin-prerender` или `prerender-spa-plugin` (запасной план) ⭐**
- Универсальный плагин
- Работает с Vite через адаптер
- **Преимущество:** Запускает локальный сервер + безголовый Chrome
- **Преимущество:** Открывает страницы и сохраняет HTML (работает тупее, но надёжнее)
- **Преимущество:** Проще внедрить в уже готовый сложный проект
- **Преимущество:** Не требует переписывания `main.tsx` под гидратацию SSG

**Вариант 3: Кастомный скрипт (Puppeteer/Playwright)**
- Полный контроль
- Больше кода
- Работает как Вариант 2, но вручную

**Рекомендация:** Начать с Варианта 1, если начнёт капризничать с React Router — переключиться на Вариант 2.

## ⚠️ КРИТИЧЕСКИ ВАЖНО: Проблема `window is not defined`

### ⚠️ Главный риск при SSG

**Проблема:**
Во время билда (`npm run build`) SSG-плагин запускает React-код в среде **Node.js**, чтобы сгенерировать HTML. В Node.js **нет объекта `window`** и **нет объекта `Telegram`**.

Если в коде есть обращения к `window` или `Telegram` на верхнем уровне (не в `useEffect`), билд упадёт с ошибкой:

```typescript
// ❌ НЕПРАВИЛЬНО - упадёт при SSG билде
const tg = window.Telegram.WebApp; // 💥 БУМ! Ошибка сборки

// ❌ НЕПРАВИЛЬНО - упадёт при SSG билде
if (window.innerWidth > 768) { ... }
```

### ✅ Решение

**1. Проверка `typeof window !== 'undefined'`:**
```typescript
// ✅ ПРАВИЛЬНО для SSG
const tg = typeof window !== 'undefined' ? window.Telegram?.WebApp : null;

// ✅ ПРАВИЛЬНО для SSG
const isMobile = typeof window !== 'undefined' ? window.innerWidth < 768 : false;
```

**2. Использование `useEffect` (не запускается при SSG генерации):**
```typescript
// ✅ ПРАВИЛЬНО для SSG
useEffect(() => {
  if (typeof window !== 'undefined' && window.Telegram) {
    const tg = window.Telegram.WebApp;
    // логика
  }
}, []);
```

**3. Использование условного рендеринга:**
```typescript
// ✅ ПРАВИЛЬНО для SSG
{typeof window !== 'undefined' && <TelegramComponent />}
```

### 📋 Чеклист перед SSG

Перед внедрением SSG нужно проверить:

- [ ] Все обращения к `window` обёрнуты в `typeof window !== 'undefined'`
- [ ] Все обращения к `Telegram` обёрнуты в проверки
- [ ] Все обращения к `document` обёрнуты в проверки
- [ ] Нет глобальных инициализаций, которые используют `window`
- [ ] Все хуки, использующие `window`, находятся в `useEffect`
