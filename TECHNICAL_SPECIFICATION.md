## 🎯 Общая информация

**Название проекта:** Sdadim DGT Prep  
**Тип приложения:** Progressive Web App (PWA) для подготовки к теоретическому экзамену DGT

#### Ключевые архитектурные решения

1. **Lazy Loading**
   - Landing страница загружается БЕЗ React Query/Supabase
   - App страницы загружаются через `React.lazy()`
   - Админ-панель полностью lazy-loaded

2. **Offline-First**
   - Service Worker кэширует статику (Workbox)
   - React Query персистит данные в IndexedDB (7 дней)
   - Offline queue для отложенных мутаций

3. **Code Splitting**
   - Vendor chunks разделены (vendor, tiptap, charts, icons)
   - Route-based splitting через React.lazy()
   - Manual chunks для изолированных библиотек

4. **State Management**
   - **TanStack Query**: серверное состояние + кэш
   - **Zustand**: локальное UI состояние (модалки)
   - **React Context**: User, Language, Theme

#### Игровая механика
- `duels` - дуэли между игроками
- `duel_players` - участники дуэлей
- `duel_questions` - вопросы в дуэли
- `duel_answers` - ответы игроков
- `race_sessions` - сессии игры "Гонка"
- `challenge_bank` - банк ежедневных заданий

#### Монетизация
- `pricing_packages` - пакеты покупок
- `premium_keys` - ключи активации премиума
- `partner_codes` - партнерские коды
- `referrals` - реферальная система

#### Прогресс и достижения
- `user_progress` - прогресс по темам
- `test_sessions` - сессии тестирования
- `achievements` - достижения
- `cosmetics` - косметика (аватары, рамки)
- `boost_inventory` - инвентарь бустеров

#### Сезоны и события
- `seasons` - игровые сезоны
- `season_progress` - прогресс по сезонам
- `daily_bonus` - ежедневные бонусы

### Row Level Security (RLS)

Все таблицы защищены RLS политиками:
- Пользователи видят только свои данные
- Публичные данные (вопросы, материалы) доступны всем
- Админы имеют расширенные права через `has_role()` функцию

### Индексы

Оптимизированы для частых запросов:
- `profiles.telegram_id` - быстрый поиск по Telegram ID
- `duels.code` - поиск дуэлей по коду
- `questions_new.topic_id` - фильтрация вопросов
- Composite индексы для сложных запросов

## 📱 PWA & Offline Support

### Service Worker (Workbox)

**Стратегии кэширования:**

| Ресурс | Стратегия | TTL |
|--------|-----------|-----|
| JS/CSS | CacheFirst | 30 дней |
| Images | CacheFirst | 7 дней |
| Supabase Storage | CacheFirst | 30 дней |
| Fonts | CacheFirst | 1 год |
| JSON (materials) | NetworkFirst | 7 дней |

**Precache:** 117 файлов (~5.7 МБ)

### React Query Persistence

- **Хранилище:** IndexedDB (через `idb-keyval`)
- **TTL:** 7 дней
- **Stale Time:** 5 минут
- **Фильтрация:** Сохраняются только стабильные данные (не realtime)

### Offline Queue

- Отложенные мутации сохраняются в IndexedDB
- Автоматическая синхронизация при восстановлении соединения
- Индикатор статуса соединения

## ⚠️ Потенциальные проблемы и рекомендации для экспертов

> **📌 Примечание:** Этот раздел дополнен на основе экспертного анализа. Критические проблемы выделены красным маркером 🔴.

### 🔴 1. Угроза переполнения БД (Supabase Free Tier) - КРИТИЧНО

**Проблема:** Лимит в 500 МБ для базы данных критически мал для геймифицированного приложения.

**Почему:** Таблицы `transactions`, `test_sessions`, `duel_questions`, `user_progress` растут экспоненциально. Логирование каждого ответа в тесте или каждого действия в дуэли быстро забьет 500 МБ.

**Риск:** Остановка сервиса через 1-2 месяца активного использования.

**Решение:**
- Реализовать автоматическую архивацию данных старше 90 дней
- Хранить только агрегированные данные (статистика вместо полных логов)
- Настроить мониторинг размера БД с алертами при 80% заполнении
- Планировать переход на Supabase Pro ($25/мес) при достижении 400 МБ

### 🔴 2. Сложность Offline-First синхронизации - КРИТИЧНО

**Проблема:** Синхронизация отложенных мутаций (Offline Queue) с Supabase при восстановлении связи.

**Почему:** 
- Если пользователь проходил тесты оффлайн неделю, а структура БД изменилась (миграция), восстановление очереди вызовет ошибки
- Возможны конфликты: пользователь потратил монеты оффлайн на одном устройстве, а потом зашел с другого и потратил их снова

**Критический сценарий:** Пользователь купил монеты на телефоне, потом зашел с планшета (где старый стейт), что-то сделал, и синхронизация планшета перезаписала баланс монет на старое значение. Пользователь потерял деньги.

**Риск:** Рассинхронизация данных, потеря прогресса, абуз валюты.

**Решение (от эксперта):**
- ✅ **Для критических полей (баланс, XP) использовать атомарные операции в SQL**
  ```sql
  -- ❌ ПЛОХО: Перезапись всего объекта (last-write-wins)
  UPDATE profiles SET coins = 150 WHERE id = 'user_id';
  
  -- ✅ ХОРОШО: Атомарная операция (решает 90% проблем)
  UPDATE profiles SET coins = coins + 10 WHERE id = 'user_id';
  ```
- Для некритических полей (имя, аватар) использовать `last-write-wins`
- Добавить версионирование кэша IndexedDB
- Валидировать совместимость данных перед синхронизацией

### 🟡 3. Латентность в дуэлях (Edge Functions)

**Проблема:** Использование Edge Functions (`duel-manager`) для игровой логики реального времени.

**Почему:** Edge Functions — это HTTP-запросы (Stateless). Для пошаговой дуэли это допустимо, но задержка на "холодный старт" функции + сетевой RTT может делать UI "дерганым".

## 💬 Ответы на вопросы эксперта

### 🏛 Архитектура и Offline-First

#### 1. Конфликты данных (write-after-write)

**Текущая реализация:**
- ❌ **Нет явной стратегии разрешения конфликтов**
- Offline queue использует `last-write-wins` неявно (последний запрос перезаписывает)
- Нет механизма merge для сложных конфликтов

**Проблема:** Если пользователь изменил профиль оффлайн на двух устройствах, последняя синхронизация перезапишет первую без уведомления.

**Критический сценарий:** Пользователь купил монеты на телефоне, потом зашел с планшета (где старый стейт), что-то сделал, и синхронизация планшета перезаписала баланс монет на старое значение. Пользователь потерял деньги.

**Рекомендация (от эксперта):**
- ✅ **Для критических полей (баланс, XP) использовать атомарные операции в SQL**
  ```sql
  -- ❌ ПЛОХО: Перезапись всего объекта
  UPDATE profiles SET coins = 150 WHERE id = 'user_id';
  
  -- ✅ ХОРОШО: Атомарная операция
  UPDATE profiles SET coins = coins + 10 WHERE id = 'user_id';
  ```
- Это решает 90% проблем синхронизации без сложного UI конфликтов
- Для некритических полей (имя, аватар) можно использовать `last-write-wins`
- Реализовать версионирование данных (оптимистичная блокировка) для сложных случаев

#### 2. Миграции и кэш IndexedDB

**Текущая реализация:**
- ❌ **Нет механизма миграции клиентского кэша**
- React Query персистит данные в IndexedDB с ключом `SDADIM_REACT_QUERY_OFFLINE_CACHE`
- При изменении схемы БД старые данные в IndexedDB могут стать невалидными

**Проблема:** Если структура данных изменилась (например, поле `coins` стало `balance`), старый кэш вызовет ошибки при десериализации.

**Рекомендация:**
- Добавить версионирование кэша (например, `SDADIM_REACT_QUERY_OFFLINE_CACHE_v2`)
- При деплое новой версии очищать старый кэш
- Использовать миграции для преобразования старых данных

#### 3. iOS Safari квоты и ITP

**Текущая реализация:**
- ⚠️ **Не проверено на практике**
- Service Worker настроен с учетом iOS ограничений (maxEntries уменьшены)
- IndexedDB используется через `idb-keyval` с fallback на memory-only

**Проблема:** iOS Safari может очистить кэш через 7 дней бездействия (ITP - Intelligent Tracking Prevention).

**Рекомендация:**
- Мониторить размер кэша и предупреждать пользователей
- Реализовать graceful degradation при очистке кэша
- Рассмотреть использование Server-Sent Events для критических данных

#### 4. Стратегия очистки (500 MB лимит)

**Текущая реализация:**
- ❌ **Нет автоматической архивации**
- Таблицы `test_sessions`, `transactions`, `duel_answers` растут без ограничений

**Рекомендация (от эксперта):**
- ✅ **Использовать `pg_cron` (Supabase Extensions) для автоматической очистки**
  ```sql
  -- Создать задачу для очистки старых данных
  SELECT cron.schedule(
    'cleanup-old-sessions',
    '0 2 * * *', -- Каждый день в 2:00
    $$
    DELETE FROM test_sessions 
    WHERE status = 'started' 
    AND created_at < NOW() - INTERVAL '24 hours';
    
    DELETE FROM transactions 
    WHERE created_at < NOW() - INTERVAL '3 months';
    $$
  );
  ```
- Хранить только агрегированные данные (статистика вместо полных логов)
- Настроить мониторинг размера БД с алертами при 80% заполнении

### 🎮 Геймплей и защита от читеров

#### 6. Валидация тестов

**Текущая реализация:**
- ⚠️ **Частичная защита**
- Idempotency через `session_id` (предотвращает повторную обработку)
- Проверка hard limits (минимальное время, количество вопросов)
- Функция `detectAbusePattern` для выявления подозрительной активности

**Проблема:**
- ❌ Нет валидации времени начала/окончания теста на сервере
- ❌ Клиент отправляет `test_duration_seconds` - можно подделать
- ⚠️ Проверка hard limits только логирует предупреждение, не блокирует
- **Критично:** Без защиты любой может написать скрипт на Python для фарма монет 24/7

**Код:**
```typescript
// supabase/functions/complete-test-and-award/index.ts
// Проверка idempotency
const { data: existingResult } = await supabase
  .from('test_results')
  .select('coins_awarded, sp_awarded')
  .eq('session_id', session_id)
  .single();

if (existingResult) {
  return { success: true, ...existingResult }; // Уже обработано
}
```

**Рекомендация (от эксперта):**

**Вариант 1: Stateful (Рекомендуется) - Запись в БД**

1. **При старте теста:**
   - Edge Function делает `INSERT` в `test_sessions` со `started_at` и `status='active'`
   - Возвращает `session_id` клиенту

2. **При завершении теста:**
   - Edge Function делает `SELECT` (проверяет время) + `UPDATE` (записывает результат)
   - Валидирует, что `NOW() - started_at >= минимальное_время`
   - Блокирует награды при подозрительной активности

**Плюсы:**
- Простота реализации и отладки
- Можно отслеживать конверсию (сколько начинают, сколько завершают)
- Абсолютная защита от подделки времени

**Минусы:**
- +1 транзакция на тест (INSERT при старте)
- Нужна очистка незавершенных сессий

**Оптимизация:**
```sql
-- Очистка незавершенных сессий через 24 часа (pg_cron)
DELETE FROM test_sessions 
WHERE status = 'started' 
AND created_at < NOW() - INTERVAL '24 hours';
```

**Вариант 2: Stateless (Альтернатива) - JWT токены**

1. **При старте теста:**
   - Edge Function НЕ пишет в БД
   - Создает JWT токен с `started_at`, `user_id`, `test_id`
   - Подписывает секретным ключом
   - Возвращает токен клиенту

2. **При завершении теста:**
   - Клиент отправляет результаты + токен
   - Edge Function проверяет подпись (0 запросов в БД)
   - Извлекает `started_at` из токена
   - Валидирует время и записывает результат

**Плюсы:**
- 0 лишних записей в БД при старте
- Нет мусорных записей для брошенных тестов

**Минусы:**
- Сложнее в реализации (криптография)
- Нельзя защититься от replay attack без БД
- Токен валиден только пару минут

**Итоговая рекомендация:** ✅ **Вариант 1 (Stateful)** - проще, надежнее, нагрузка ничтожна для текущих масштабов

**Анализ нагрузки на БД:**

**Текущая реализация (без защиты):**
- 1 транзакция на тест (только завершение)
- **Проблема:** Можно отправлять 100 запросов/сек → 100 записей в БД → миллион монет

**С защитой (Stateful):**
- 2 транзакции на тест (старт + завершение)
- **Защита:** При попытке абуза - только легкая проверка (SELECT), тяжелая запись (INSERT) отменяется
- **Нагрузка:** Для PostgreSQL это "комариный укус" - даже на Free Tier сотни инсертов/сек не проблема

**Вывод:** Платим одним лишним INSERT (миллисекунды), чтобы сэкономить гигабайты и тысячи долларов в будущем.

## 📋 План действий (приоритизированный)

### Неделя 1 (Критично) - Security First

#### 1. ✅ **Переместить daily bonus в Edge Function**

**Текущее состояние:**
- ❌ Логика на клиенте (`src/pages/Index.tsx`, `src/pages/DailyBonus.tsx`)
- ❌ Используется клиентское время: `new Date().toISOString().split('T')[0]` (локальное, не UTC)
- ❌ НЕТ Edge Function `claim-daily-bonus`
- ⚠️ Можно обойти через прямой API вызов или изменение времени на устройстве

**Что нужно сделать:**
- ✅ Создать Edge Function `claim-daily-bonus`
- ✅ Использовать серверное время UTC (не клиентское)
- ✅ Добавить идемпотентность через `user_id + date` (UTC)
- ✅ **Важно:** Привязка к UTC 00:00 (не к локальному времени пользователя)
  - Если у пользователя 23:59 локально, а на сервере 01:00 следующего дня — это новый день
  - UTC проще и безопаснее, чем часовые пояса

#### 2. ✅ **Добавить валидацию времени тестов на сервере**

**Текущее состояние:**
- ✅ Есть функция `checkHardLimits` в `complete-test-and-award/index.ts`
- ✅ Проверяет минимальное время: `minDuration = config.minTestDurationBase + Q * config.minTestDurationPerQuestion`
- ❌ **НО:** Проверка только логирует предупреждение, НЕ блокирует награды (строка 418: "Не блокируем")
- ❌ НЕТ записи `started_at` в `test_sessions` при старте теста
- ❌ НЕТ валидации времени на сервере (используется клиентское `test_duration_seconds`)
- ⚠️ Можно подделать время, отправив `test_duration_seconds` с клиента

**Что нужно сделать:**
- ✅ Реализовать **Stateful валидацию** (Вариант 1)
- ✅ При старте теста: `INSERT` в `test_sessions` со `started_at` и `status='active'`
- ✅ При завершении: валидация времени на сервере и **блокировка наград** при подозрении
- ✅ **Критично:** Проверка **минимального** времени прохождения
  - Если тест из 30 вопросов решен за 10 секунд — это бот
  - Даже если `started_at` был 10 минут назад (скрипт мог ждать)
  - Человек не может читать быстрее определенной скорости
  - Формула: `questions_count * MIN_TIME_PER_QUESTION` (например, 3 сек/вопрос)
- ✅ Настроить `pg_cron` для очистки незавершенных сессий (> 24 часов)

3. ✅ **Написать E2E тесты для платежей**
   - Тесты для всех платежных шлюзов (Stars, Stripe, Cryptomus)
   - Тесты для edge cases (webhook failures, retries)

### Неделя 2 (Высокий приоритет) - Data & Sync
4. ✅ **Реализовать атомарные операции для критических полей**
   - Заменить `UPDATE profiles SET coins = 150` на `UPDATE profiles SET coins = coins + 10`
   - Применить для всех операций с балансом, XP, SP
   - Это решит 90% проблем синхронизации
   - **Реализация через RPC функции (рекомендуется):**
     ```sql
     CREATE OR REPLACE FUNCTION add_coins(
       p_user_id UUID, 
       p_amount INTEGER
     ) RETURNS VOID AS $$
     BEGIN
       UPDATE profiles 
       SET coins = coins + p_amount 
       WHERE id = p_user_id;
     END;
     $$ LANGUAGE plpgsql SECURITY DEFINER;
     ```
   - Вызывать через `supabase.rpc('add_coins', { p_user_id: userId, p_amount: 10 })`
   - Это безопаснее, чем прямой `update({ coins: 150 })`

5. ✅ **Настроить автоматическую архивацию через pg_cron**
   - Создать cron-задачи для очистки старых данных
   - `test_sessions` (статус 'started' > 24 часов)
   - `transactions` (> 3 месяцев)
   - `duel_answers` (> 90 дней)

6. ✅ **Добавить conflict resolution для offline queue**
   - Для критических полей использовать атомарные операции (см. пункт 4)
   - Для некритических - `last-write-wins` с уведомлением пользователя

7. ✅ **Настроить мониторинг размера БД**
   - Алерты при 80% заполнении
   - Дашборд для отслеживания роста таблиц
