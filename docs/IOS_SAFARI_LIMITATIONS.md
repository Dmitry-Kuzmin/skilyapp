# iOS Safari PWA Limitations

## 🚨 Критические ограничения iOS Safari

Apple **специально ограничивает** PWA на iPhone, чтобы продвигать нативные приложения из App Store.

---

## 📱 Что происходит на iPhone

### Проблема, которую наблюдал Дима:

```
1. Открыл приложение С интернетом → работает ✅
2. Выключил интернет → работает 2-3 минуты ✅
3. Через 2-3 минуты → "TypeError: Load failed" ❌
4. Reload → "Safari не удалось открыть страницу" ❌
```

### Причина:

**iOS Safari агрессивно очищает Service Worker кэш!**

---

## 🗑️ Когда iOS Safari чистит кэш

### 1. По времени (самое частое)

- **2-7 дней** бездействия → кэш удаляется полностью
- **Неделя** без открытия сайта → SW деактивируется

### 2. По памяти

- **>50 МБ** кэша на один домен → начинает чистить
- **Общая нехватка памяти** на устройстве → чистит агрессивно
- **Фоновые вкладки** → чистит в первую очередь

### 3. При событиях

- **Закрытие Safari** → может деактивировать SW
- **Переход в другое приложение** → может чистить кэш
- **Низкий заряд батареи** → чистит кэш для экономии

### 4. Специальные режимы

- **Private Browsing** → SW вообще не работает
- **Low Power Mode** → агрессивная очистка
- **Cellular only** (без Wi-Fi) → ограничивает кэш

---

## 📊 Лимиты iOS Safari

| Ресурс | Desktop Safari | iOS Safari | Разница |
|--------|----------------|------------|---------|
| **Cache Storage** | ~500 МБ | ~50 МБ | **10x меньше** |
| **IndexedDB** | ~1 ГБ | ~50 МБ | **20x меньше** |
| **Service Worker lifetime** | Постоянный | Временный | Деактивируется |
| **Background Sync** | Работает | ❌ Не работает | Недоступен |
| **Push Notifications** | Работает | ❌ Только из App Store | Недоступен |

---

## 🛠 Что мы сделали для iOS

### 1. Уменьшили precache

**Было:**
```
globPatterns: ['**/*.{js,css,html,ico,png,svg,woff,woff2,json}']
→ 118 файлов (~5.7 МБ)
```

**Стало:**
```
globPatterns: ['**/*.{js,css,html,ico,svg}']
globIgnores: ['**/data/materials/**', '**/*.{png,jpg,jpeg}']
→ ~40-50 файлов (~3-4 МБ)
```

Изображения и JSON загружаются **runtime** (по запросу).

### 2. Уменьшили runtime cache limits

| Кэш | Было | Стало | Экономия |
|-----|------|-------|----------|
| **app-static-assets** | 150 | 80 | -47% |
| **supabase-storage** | 300 | 100 | -67% |
| **images** | 200 | 60 | -70% |
| **local-data** | 500 | 100 | -80% |

### 3. Сократили TTL

| Кэш | Было | Стало |
|-----|------|-------|
| **images** | 30 дней | 7 дней |
| **local-data** | 30 дней | 7 дней |

### 4. NetworkFirst для JSON

Вместо `CacheFirst` для JSON используем `NetworkFirst`:
- Сначала пытается загрузить с сети
- Если нет сети (3 сек timeout) → fallback на кэш
- Меньше устаревших данных

## ⚠️ Известные ограничения

### 1. Холодный старт без интернета

**НЕ работает** если:
- Вообще никогда не открывал с интернетом
- iOS почистил весь кэш
- Safari в Private mode

**Работает** если:
- Недавно открывал (< 2-7 дней)
- Есть precache

**Вероятность:** ~40-60% на iOS

### 2. Долгая работа offline

**Работает:** 2-3 минуты гарантированно ✅
**Может сломаться:** После 5-10 минут offline ⚠️
**Точно не работает:** Через неделю offline ❌

**Причина:** iOS может чистить кэш фоново.

### 3. Фоновые вкладки

Если Safari в фоне (другое приложение открыто):
- iOS может деактивировать SW
- Кэш может быть почищен
- При возврате может потребоваться reload с интернетом

## 🎯 Рекомендации для пользователей

### Для лучшего offline опыта:

**1. Добавить на Home Screen (A2HS):**

```
Safari → Share → Add to Home Screen
```

**Почему:** 
- iOS **меньше** чистит кэш у "установленных" PWA
- Fullscreen режим (без Safari UI)
- Отдельный процесс (лучшая изоляция)

**2. Открывать хотя бы раз в неделю:**

iOS чистит кэш после **2-7 дней** бездействия.

**3. Держать заряд > 20%:**

Low Power Mode агрессивно чистит кэши.

## 🔧 Альтернативы для iOS

### Вариант 1: Уменьшить кэш ещё больше

**Плюсы:**
- iOS будет реже чистить
- Дольше работает offline

**Минусы:**
- Меньше данных offline
- Больше запросов к серверу

### Вариант 2: Нативное iOS приложение

**Плюсы:**
- Полный контроль над кэшем
- Push notifications
- Background sync
- Нет лимитов Safari

**Минусы:**
- Нужно писать Swift/React Native
- App Store review
- Поддержка двух кодовых баз

### Вариант 3: Hybrid (Capacitor/Ionic)

**Плюсы:**
- Та же React кодовая база
- Native capabilities
- Лучше чем Safari PWA

**Минусы:**
- Дополнительная настройка
- App Store deployment

## 📝 Workarounds (что мы применили)

### 1. Минимизация precache

Кэшируем **только критичные** файлы:
- ✅ index.html
- ✅ vendor.js, index.js (main chunks)
- ✅ CSS
- ✅ Fonts, icons
- ❌ Images (runtime)
- ❌ JSON files (runtime)

### 2. Runtime cache с лимитами

Вместо "кэшировать всё" → кэшируем **по запросу**:
- Открыл страницу → JS/CSS закэшировались
- Посмотрел картинку → закэшировалась
- Прочитал materials → JSON закэшировался

**Преимущество:** Кэш растёт постепенно, iOS терпит.

### 3. NetworkFirst для данных

Не полагаемся **только** на кэш:
- Сначала пробуем сеть (3 сек)
- Нет сети → кэш
- Есть сеть → свежие данные + обновляем кэш

**Преимущество:** Актуальные данные, но offline fallback есть.

### 4. React Query для важных данных

Критичные данные (профиль, прогресс, dashboard):
- Хранятся в **IndexedDB** (отдельно от SW)
- **7 дней** retention
- Не зависят от SW

**Преимущество:** Даже если iOS убил SW, данные остались.

## 🧪 Тестирование на iOS

### Проверка кэша:

```javascript
// В Safari Console (Remote Debugging)
caches.keys().then(keys => {
  console.log('Total caches:', keys.length);
  
  keys.forEach(async (key) => {
    const cache = await caches.open(key);
    const requests = await cache.keys();
    console.log(key, ':', requests.length, 'entries');
  });
});
```

### Проверка размера:

```javascript
// Примерная оценка
navigator.storage.estimate().then(estimate => {
  const used = (estimate.usage / 1024 / 1024).toFixed(2);
  const quota = (estimate.quota / 1024 / 1024).toFixed(2);
  console.log(`Used: ${used} MB / ${quota} MB`);
});
```

**Если > 40 МБ** → iOS скоро почистит.

## 💡 Рекомендации

### Для разработчиков:

1. **Минимизировать precache** (< 5 МБ)
2. **Runtime cache по запросу** (не всё сразу)
3. **maxEntries** держать низкими (< 100 на кэш)
4. **Критичные данные** → IndexedDB, не SW cache
5. **Monitoring:** отслеживать когда iOS чистит кэш

### Для пользователей:

1. **Add to Home Screen** → лучший offline опыт
2. **Открывать хотя бы раз в неделю**
3. **Избегать Low Power Mode** при offline работе
4. **Держать заряд > 20%**

### Для product owners:

1. **Онбординг:** "Для offline режима добавьте на Home Screen"
2. **Expectations:** Объяснять что offline "до 7 дней", не "навсегда"
3. **Fallback UX:** Красивый экран "Reconnect to continue"
4. **Metrics:** Отслеживать % пользователей с активным SW

## 🎯 Наш подход (компромисс)

### Что работает:

✅ **Desktop** - полноценный offline (недели)
✅ **Android** - хороший offline (дни)
✅ **iOS (A2HS)** - приемлемый offline (дни)
⚠️ **iOS Safari** - короткий offline (минуты-часы)

### Стратегия:

1. **Делаем PWA максимально лёгким** (< 20 МБ общего кэша)
2. **Критичные данные** в IndexedDB (держатся дольше)
3. **Graceful degradation** (если кэша нет → показываем понятное сообщение)
4. **Рекомендуем A2HS** для частых пользователей
