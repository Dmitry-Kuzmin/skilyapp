# ⚡ Оптимизация расписания cron для сезонов

## 🤔 Почему я сделал ежедневную проверку?

Изначально я настроил ежедневную проверку (`0 0 * * *`) для **надёжности**:
- Чтобы не пропустить момент окончания сезона
- На случай, если сезон закончится не точно в полночь
- Для отладки и тестирования

Но ты прав — **это избыточно** для сезонов, которые длятся 28 дней!

---

## 💰 Стоимость pg_cron в Supabase

**Хорошие новости:**
- ✅ **pg_cron бесплатный** — нет дополнительной платы за расширение
- ✅ Доступен на всех планах Supabase (включая Free)

**Но важно:**
- ⚠️ Задачи потребляют ресурсы БД (CPU, память)
- ⚠️ Частые задачи могут замедлить базу данных
- ⚠️ На Free плане ограниченные ресурсы

**Вывод:** Лучше оптимизировать расписание, чтобы не нагружать БД без необходимости.

---

## 📅 Оптимальное расписание для сезонов

Сезоны длятся **28 дней (4 недели)**, поэтому оптимальные варианты:

### Вариант 1: Раз в день (рекомендуется)
```sql
'0 0 * * *'  -- Каждый день в полночь UTC
```
**Плюсы:**
- Надёжно — не пропустит окончание сезона
- Минимальная нагрузка (1 раз в день)
- Быстро обработает сезон в день окончания

**Минусы:**
- Выполняется даже когда сезонов нет

### Вариант 2: Раз в неделю (оптимально)
```sql
'0 0 * * 0'  -- Каждое воскресенье в полночь UTC
```
**Плюсы:**
- Меньше нагрузка на БД
- Достаточно для сезонов (28 дней)
- Обработает сезон в течение недели после окончания

**Минусы:**
- Может быть задержка до 7 дней (но это нормально)

### Вариант 3: Раз в месяц (минимально)
```sql
'0 0 1 * *'  -- Первое число каждого месяца в полночь UTC
```
**Плюсы:**
- Минимальная нагрузка
- Подходит, если сезоны всегда заканчиваются в конце месяца

**Минусы:**
- Может быть задержка, если сезон закончился не в конце месяца

## 🔄 Как изменить расписание существующей задачи

### Способ 1: Удалить и создать заново (проще)

```sql
-- 1. Удалить старую задачу
SELECT cron.unschedule('daily-season-rewards-check');

-- 2. Создать новую с оптимальным расписанием (раз в неделю)
SELECT cron.schedule(
  'weekly-season-rewards-check',
  '0 0 * * 0', -- Каждое воскресенье в полночь UTC
  $$
  SELECT mark_season_for_rewards_distribution();
  $$
);
```

### Способ 2: Изменить через alter_job (если доступно)

```sql
-- Изменить расписание существующей задачи
SELECT cron.alter_job(
  jobid := 1,  -- ID твоей задачи
  schedule := '0 0 * * 0'  -- Новое расписание (каждое воскресенье)
);
```

## 🎯 Рекомендация

**Для сезонов длительностью 28 дней рекомендую:**

```sql
-- Удалить ежедневную задачу
SELECT cron.unschedule('daily-season-rewards-check');

-- Создать еженедельную (каждое воскресенье)
SELECT cron.schedule(
  'weekly-season-rewards-check',
  '0 0 * * 0', -- Каждое воскресенье в полночь UTC
  $$
  SELECT mark_season_for_rewards_distribution();
  $$
);
```

**Почему это оптимально:**
- ✅ Сезон длится 28 дней — проверка раз в неделю достаточна
- ✅ Минимальная нагрузка на БД
- ✅ Обработает сезон в течение недели после окончания
- ✅ Бесплатно и эффективно

## 📊 Сравнение вариантов

| Расписание | Частота | Нагрузка | Задержка обработки | Рекомендация |
|------------|---------|----------|-------------------|--------------|
| `0 0 * * *` | Каждый день | Средняя | 0-1 день | ✅ Хорошо для тестирования |
| `0 0 * * 0` | Раз в неделю | Низкая | 0-7 дней | ✅ **Оптимально для продакшена** |
| `0 0 1 * *` | Раз в месяц | Минимальная | 0-30 дней | ⚠️ Только если сезоны всегда в конце месяца |

## ✅ Итог

1. **pg_cron бесплатный**, но потребляет ресурсы БД
2. **Ежедневная проверка избыточна** для 28-дневных сезонов
3. **Рекомендую еженедельную проверку** (`0 0 * * 0`)
4. Можно изменить расписание через `cron.unschedule()` + `cron.schedule()`
