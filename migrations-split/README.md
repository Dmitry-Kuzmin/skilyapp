# Инструкции по применению миграций

## Вариант 1: Применение по частям (рекомендуется)

Применяйте файлы PART_XX.sql по порядку через Supabase SQL Editor:

1. **PART_01.sql** - миграции 1-10 (создание базовых таблиц)
2. **PART_02.sql** - миграции 11-20 (расширение функционала)
3. **PART_03.sql** - миграции 21-30 (уведомления и realtime)
4. **PART_04.sql** - миграции 31-40 (race game и road race)
5. **PART_05.sql** - миграции 41-50 (learning map система)
6. **PART_06a_add_editor_enum.sql** - добавление enum значения 'editor' (ВАЖНО: применить ПЕРВЫМ)
7. **PART_06b_update_policies.sql** - обновление RLS политик для редакторов (применить ПОСЛЕ PART_06a)

## Порядок применения

1. Откройте Supabase Dashboard → SQL Editor
2. Примените PART_01.sql
3. Проверьте, что нет ошибок
4. Примените PART_02.sql
5. И так далее по порядку
6. **ВАЖНО для PART_06**: 
   - Сначала примените **PART_06a_add_editor_enum.sql** (добавление enum)
   - Затем примените **PART_06b_update_policies.sql** (обновление политик)
   - Это необходимо, так как в PostgreSQL нельзя использовать новое значение enum в той же транзакции

## Если возникла ошибка

Если при применении какой-то части возникла ошибка:

1. Запомните, в какой части (PART_XX.sql) произошла ошибка
2. Откройте этот файл
3. Найдите проблемную миграцию (поиск по тексту ошибки)
4. Примените миграции до проблемной вручную
5. Исправьте проблемную миграцию
6. Продолжите с остальными

## Примечания

- Все миграции содержат проверки `IF NOT EXISTS`, поэтому их можно применять повторно
- Если миграция уже применена, она пропустится
- Рекомендуется применять по одной части за раз и проверять результат

## Альтернативный способ

Если хотите применить все сразу, используйте исходный файл `ALL_MIGRATIONS_SAFE.sql`, но это может вызвать проблемы с большим размером файла.
